```
sample.pdf
  └─ pdf_to_json.py   →  sample_raw.json       (pages 배열, 페이지 단위)
```

## 2. `json_structurer.py` 버그 진단 및 수정 명세

### 2-1. 진단된 버그 목록

| # | 함수/위치 | 버그 내용 | 영향 |
|---|----------|----------|------|
| **BUG-S1** | `process()` — 사업 경계 감지 | `'사업명' in text_no_space` 조건이 **`내역사업명`** 등 부분 문자열에도 매칭됨 | 2페이지(`가. 예산 총괄표` 헤더의 `사업명`), 4페이지(`내역사업명` 컬럼) 등 **오탐** 발생 |
| **BUG-S2** | `process()` — 사업 경계 감지 | 오탐 페이지에서 `p_id`가 `None`이어서 분배는 막히지만, `current_section`이 이미 `"projects"`인 경우 **이전 사업의 ID로 데이터가 계속 쌓임** → 정상 동작처럼 보이나, 다음 사업 코드를 감지하는 `p_id` 검사 자체가 본문의 코드 숫자(`123,339` 등)와 충돌 가능 | 다중 사업 처리 시 경계 오탐 |
| **BUG-S3** | `_get_project_id()` | `r'(\d{4}-\d{3})'` 패턴이 본문의 **예산 금액·연락처** 등 4자리-3자리 숫자를 코드로 오인할 수 있음 | 드문 케이스이나 실제 데이터에서 발생 가능 |
| **BUG-S4** | `save()` — multi 모드 | `projects` 파트 저장 시 `ProjectData` dict가 그대로 직렬화되어 **Pydantic 모델 내부 구조**(`.dict()`)와 불일치 | multi 모드 저장 파일이 파서에서 읽히지 않을 수 있음 |
| **BUG-S5** | `process()` — Tail 판정 | `elif current_section != "tail":` 분기 때문에 같은 페이지에서 Tail 키워드 감지 후에도 그 아래 코드에서 Project 판정 로직이 **실행되지 않음** — 이건 정상이지만, Tail 키워드가 사업 페이지 중간에 나오면 해당 페이지 데이터가 Tail에 분류됨 | 경계 설정 주의 필요 |

## 3. `budget_parser.py` 전체 필드 추출 명세

> 기준: `sample_structured.json` → 각 project의 `text` + `tables` 배열  
> v6에서 수정 완료된 항목 포함하여 전체 정리

### 3-1. 입력 데이터 구조 이해

`sample_structured.json` 기준 `tables` 배열 순서 (사업마다 다를 수 있음):

| 인덱스 | 표 제목 | 주요 컬럼 |
|--------|--------|---------|
| `[0]` | 사업명 표 | `사 업 명` / `(N) 사업명 (코드)` |
| `[1]` | 사업 코드 정보 | 회계, 소관, 실국, 분야, 부문 |
| `[2]` | 프로그램/단위/세부사업 | 코드, 명칭 |
| `[3]` | 사업 성격 | 신규, 계속, 완료 + ○ 마킹 |
| `[4]` | 지원 형태 및 지원율 | 직접, 출자, 출연, 보조, 융자, 국고보조율, 융자율 |
| `[5]` | 소관부처 및 시행주체 | 사업명, 구분(소관부처/시행주체), 기관명 |
| `[6]` | 예산 총괄표 | 사업명, 2024결산, 2025본예산, 추경(A), 요구안, 본예산(B), 증감 |
| `[7]` | 기능별(내역사업별) 내역 | 2024/2025 예산액/집행/이월/불용, 2026예산 |
| 이후 | 지원 비율 및 법적근거, 집행절차, 성과지표, 결산 등 | — |

> ⚠️ 인덱스는 가변적. 반드시 **헤더 키워드로 식별**해야 함.

### 3-2. 필드별 추출 명세 전체표

#### A. 기본 식별 정보

| 필드 | 타입 | 소스 | 추출 방식 | 정규식 / 로직 |
|------|------|------|----------|-------------|
| `code` | str | `tables[0]` 우선, text fallback | tables[0][1][0] 셀에서 `(NNNN-NNN)` 추출 | `r'\((\d{4}-\d{3,4})\)'` |
| `project_name` | str | 동일 | 코드 앞 텍스트, 순번 `(N)` 제거 | `re.sub(r'^\(\d+\)\s*', '', name_part)` |
| `name` | str | 파생 | `{department}_{project_name}` | config `derived_fields` |
| `page_start` | int | text | 첫 번째 유효 페이지 마커 | `r'-\s*(\d{2,6})\s*-'` + 필터 `100~9999 & not(2000~2030)` |
| `page_end` | int | text | `다. 최근 4년간 결산내역` **이전** 텍스트의 마지막 마커 | 동일 패턴, cutoff 적용 |

> ⚠️ **페이지 마커 필터**: `-2024-` `-2025-` 등 연도값 오인식 방지  
> → `2000 ≤ v ≤ 2030` 범위는 연도로 간주하고 **제외**

#### D. 예산 정보 (`tables[6]` — 예산 총괄표)

**헤더 식별:** 2행에 걸쳐 있는 2단 헤더 → row[0]+row[1] 합산으로 컬럼 레이블 구성

```
row[0] = ["사업명", "2024년\n결산", "2025년 예산", null,     "2026년 예산", null,     "증감...", null]
row[1] = [null,    null,          "본예산",      "추경(A)", "요구안",     "본예산(B)", null,    "(B-A)/A"]
합산   =  "사업명"  "2024년결산"   "2025년예산본예산" "추경(A)" "요구안"    "2026년예산본예산(B)"  ...
```

| 필드 | 컬럼 레이블 패턴 (느슨한 매칭) | 비고 |
|------|--------------------------|------|
| `budget.2024_settlement` | `r'2024.*결산'` or `r'결산'` | |
| `budget.2025_original` | `r'2025.*본예산(?!.*추경)'` or `r'본예산(?!\(B\))'` | 추경(A) 제외 |
| `budget.2025_supplementary` | `r'추경\(A\)'` or `r'추경'` | |
| `budget.2026_request` | `r'요구안'` | |
| `budget.2026_budget` | `r'본예산\(B\)'` or `r'2026.*본예산'` | |

**데이터 행 선택:**  
1순위: `합계` 또는 `계` 포함 행  
2순위: 숫자가 가장 많은 행 (fallback)

**파생 필드:**
```python
change_amount = budget.2026_budget - (budget.2025_supplementary or budget.2025_original)
change_rate   = change_amount / budget.2025_original * 100  # 0이 아닐 때
```

> ⚠️ **금액 단위**: PDF 기준 **백만원**. DB 저장 단위가 원이라면 `× 1_000_000` 필요  
> ⚠️ **△ 기호**: 감소 표시 `△` → 음수로 변환 (`re.sub(r'[△▲]', '-', val)`)

#### E. 내역사업 (`tables[7]` — 기능별 내역표)

**식별 조건:** `"2024"` AND `"2025"` AND `"2026"` AND (`"집행액"` OR `"예산액"`) 모두 포함

**셀 구조 (멀티라인):**  
col[0]에 합계+내역사업명이 `\n`으로 병합되어 있음:
```
col[0] = "○ 기능별 분류(합계)\n·중소기업 핵심인력\n성과보상기금\n·중소기업 인식개선\n및성과공유확산"
col[1] = "123,339\n121,685\n1,654"    ← 2024 예산액(추경)
col[6] = "19,347\n17,593\n1,754"     ← 2025 예산액(추경)
col[-1] = "10,440\n8,686\n1,754"    ← 2026 예산
```

**추출 로직:**
```python
all_lines = col[0].split('\n')
# 합계 행(첫 줄) 제거: "○"로 시작하거나 "합계", "분류" 포함
names = [n for n in all_lines if not re.match(r'^[○◯]|합계|분류', n)]
offset = len(all_lines) - len(names)  # 제거된 줄 수
# 각 값 컬럼도 offset 만큼 앞줄 제거 후 매핑
```

> ⚠️ **근본 원인**: PDF→JSON 추출 시 셀이 병합되어 있음. 2줄짜리 사업명(`핵심인력\n성과보상기금`)은  
> 상위 단계(pdf_to_json)에서 셀을 분리하지 않는 한 파서에서 완전 복원 불가.  
> **현실적 대응**: 앞 구두점(`·`, `·`, `ㆍ`) 기준으로 분리하고, 단어 잇기(`이전줄 + 현재줄`) 시도

#### F. 기간 정보 (text에서 정규식 추출)

| 필드 | 정규식 | 예시 원문 |
|------|--------|---------|
| `project_period.raw` | `r'사업기간\s*[:：]\s*(.+?)(?:\n|$)'` | `'05년 ~ 계속`, `2026년 ~ 2028년(3년)` |

| `project_period.duration` | 파생: `str(end - start + 1) + '년'` | end_year가 있을 때만 |

| `effectiveness` | `4) 사업효과` | `5) 타당성`, `다. 최근` | |
| `total_cost.raw` | — | — | `r'총사업비\s*(?:\(해당.*?\))?\s*[:：]\s*(.+?)(?:\n\|$)'` |

#### I. 하위 배열 데이터 (미완성 — 추가 구현 필요)

| 필드 | 소스 위치 | 구조 | 현재 상태 |
|------|---------|------|---------|
| `project_managers` | `tables[5]` 소관부처 표 | `{sub_project, managing_dept, implementing_agency, manager, phone}` | 소관부처만 추출, 담당자/전화는 null |
| `yearly_budgets` | text — `최근 5년 간 투입된 사업비` 하단 표 | `{"연도": 금액}` dict | 미구현 |
| `kpi` | `4) 사업효과` — 성과지표 표 | `{name, weight, targets, actuals, ...}` | 미구현 |
| `history` | `성과지표 이외의 연도별 사업추진 경과` | `{year, description}` 배열 | 미구현 |

**`yearly_budgets` 추출 예시 로직:**
```python
# text에서 추출
m = re.search(r'연도\s+([\d\s]+)\n사업비\s+([\d,\s]+)', text)
if m:
    years = m.group(1).split()
    amounts = re.findall(r'[\d,]+', m.group(2))
    yearly_budgets = {y: _to_float(a) for y, a in zip(years, amounts)}
```

**`history` 추출 예시 로직:**
```python
# "성과지표 이외의 연도별 사업추진 경과 및 실적" 블록
block = _extract_block(text, '성과지표 이외의 연도별', ['③ 향후', '③향후'])
# 연도 패턴: 4자리 연도가 줄 시작에 단독으로 있는 패턴
entries = re.split(r'\n(?=\d{4}\n)', block or '')
for entry in entries:
    m_yr = re.match(r'(\d{4})\n(.+)', entry, re.DOTALL)
    if m_yr:
        history.append({"year": int(m_yr.group(1)), "description": m_yr.group(2).strip()})
```

## 4. config.yaml vs template.json 대조표

### 4-1. 스키마 불일치 항목

| config.yaml `project_columns` 키 | budget dict 실제 키 | 불일치 여부 | 수정 필요 |
|----------------------------------|-------------------|-----------|---------|
| `budget_2024_settlement` | `2024_settlement` | ❌ **불일치** | parser에서 별도 매핑 필요 |
| `budget_2025_original` | `2025_original` | ❌ **불일치** | 동일 |
| `budget_2026_budget` | `2026_budget` | ❌ **불일치** | 동일 |

> v5 버그의 핵심 원인. v6에서 `budget_col_map` 딕셔너리로 분리해 해결.

### 4-2. config.yaml `defaults` 주의사항

```yaml
# ⚠️ 이 설정이 실제 추출 데이터를 덮어쓸 수 있음
effectiveness: (작성예정)  # ← 반드시 null로 변경
```

> `defaults`는 파서가 추출 실패 시에만 사용해야 함.  
> `effectiveness`에 하드코딩 문자열이 있으면 추출 성공 여부와 무관하게 덮어씌워짐.

### 4-3. template.json vs 실제 출력 대조

| template.json 필드 | parser 출력 상태 | 비고 |
|-------------------|----------------|------|
| `budget.change_amount` | ✅ v6에서 파생 계산 | `2026_budget - 2025_original` |
| `budget.change_rate` | ✅ v6에서 파생 계산 | `change_amount / 2025_original * 100` |
| `project_period.duration` | ⚠️ 종료연도가 null이면 null | `'05년 ~ 계속` 패턴 |
| `overview` 3개 키 | ✅ v6에서 추출 | |
| `yearly_budgets` | ⚠️ 미완성 | 별도 구현 필요 |
| `kpi`, `history` | ❌ 미구현 | 하위 배열 파싱 필요 |
| `sub_projects` 이름 | ⚠️ 멀티라인 병합 | pdf_to_json 레벨 문제 |
