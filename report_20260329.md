# KAIB2026 프로젝트 스크립트 구조 및 경로 검증 보고서

**작성일:** 2026-03-29

## 1. 프로젝트 구조 개요

- **preProc**: PDF → 규격화된 JSON 변환
- **pipeline**: 검증된 JSON → 통합/분석/메타데이터 처리 → `budget_db.json` 생성 및 `web/data/`로 복사
- **web**: `budget_db.json`, `analysis.json` 기반 웹서비스

---

## 2. 경로 및 스크립트 검증 결과

### 2.1 preProc
- PDF 파일을 읽어 JSON으로 변환하는 스크립트(`pdf_to_json.py`, `json_structurer.py` 등) 존재
- 입력/출력 경로가 하드코딩된 부분이 일부 존재 (예: `../database/raw/`, `../database/parse_result/` 등 상대경로)
- **개선점:**
  - 경로를 config 파일(`config.yaml`)에서 읽도록 통일 필요
  - 스크립트 실행 위치에 따라 상대경로 오류 발생 가능 → 절대경로 변환 로직 추가 권장

### 2.2 pipeline
- 데이터 병합, 분석, 메타데이터 통합 등 주요 파이프라인 스크립트(`convert.py`, `master_builder.py`, `export_a4.py` 등) 존재
- 최종 산출물(`budget_db.json`)이 `web/data/`로 복사되는 구조
- **문제점:**
  - 일부 스크립트에서 경로가 하드코딩되어 있어, 디렉터리 구조 변경 시 오류 발생 가능
  - 임시/중간 산출물 폴더(`tmp/`, `parse_result/`, `raw/`)가 명확히 구분되어 있으나, 불필요하게 중복되는 데이터가 있을 수 있음
- **개선점:**
  - 경로 및 파일명은 config에서 일괄 관리
  - 불필요한 중간 산출물 정리 및 자동 삭제 스크립트 추가

### 2.3 web
- `web/data/` 내 `budget_db.json`, `analysis.json` 등 데이터 파일 활용
- JS 코드에서 데이터 경로가 하드코딩되어 있음 (`data/budget_db.json` 등)
- **개선점:**
  - 배포 환경에 따라 데이터 경로를 동적으로 처리할 수 있도록 수정
  - 불필요한 테스트/임시 파일(`playwright_output.txt` 등) 정리

## 3. 불필요 폴더 및 파일
- `database/backup/`, `database/tmp/`, `database/parse_result/residual/` 등은 주기적으로 정리 필요
- `scripts/legacy_tools/` 내 사용하지 않는 스크립트 아카이빙 또는 삭제
- `web/playwright_output.txt`, `web/playwright-test-report.md` 등 테스트 산출물은 별도 폴더로 이동 권장

- **중간 산출물 자동 정리:** 파이프라인 완료 후 임시파일 자동 삭제

## 6. 주요 3개 파일 데이터 구조 비교 및 차이 분석

### 6.1 config/config.yaml
- **역할:** 전체 데이터 파이프라인의 경로, 필드 매핑, 기본값, 검증 규칙 등 메타 설정을 담당하는 YAML 파일
- **주요 구조:**
  - `paths`, `output`, `xlsx`, `column_mapping`, `defaults`, `validation`, `schema` 등 다양한 설정 섹션
  - 실제 데이터가 아닌, 데이터 생성/가공/검증을 위한 기준과 규칙을 정의

### 6.2 config/template.json
- **역할:** budget_db.json의 구조적 템플릿(샘플/스키마)로, 실제 데이터 없이 필드명과 타입, 예시값, 설명을 제공
- **주요 구조:**
  - `metadata`: 통계/메타정보(예시값)
  - `projects`: 실제 데이터 구조의 샘플(모든 필드, 타입, 설명 포함)
  - `analysis`: 분석 결과 구조(빈 값/예시)
- **특징:**
  - 실제 값 없이 구조와 타입, 설명만 포함(문서화 목적)
  - 신규 필드 추가/변경 시 기준점 역할

### 6.3 database/backup/budget_db.json_backup
- **역할:** 통합된 예산 데이터의 백업본 (실제 서비스/분석 대상)
- **주요 구조:**
  - `metadata`: 전체 프로젝트/부처/예산 총계 등 실제 메타정보
  - `projects`: 각 사업별 상세 데이터(수백~수천 개 오브젝트, 필드명은 template.json과 유사)
  - `analysis`: 부처별/유형별/도메인별 분석, 증감 상위, 중복 등 통계/네트워크 정보
- **특징:**
  - 실제 데이터 값이 포함되어 있음
  - 일부 필드는 config.yaml의 매핑명과 다를 수 있음(예: `project_name` vs `name`)
  - template.json의 구조와 거의 일치하나, 값/타입/필드 누락/추가 가능

### 6.4 3개 파일 구조 비교 요약
| 항목 | config.yaml | template.json | budget_db.json_backup |
|---|---|---|---|
| 역할 | 파이프라인 설정/매핑/검증 | 데이터 구조 샘플/문서화 | 실제 통합 데이터(서비스/분석) |
| 데이터 | 없음 | 없음(예시/타입/설명) | 실제 값 포함 |
| 주요 섹션 | paths, column_mapping, schema 등 | metadata, projects, analysis | metadata, projects, analysis |
| 활용 | 코드/파이프라인 기준 | 구조 변경/문서화 기준 | 서비스/분석/백업 |
| 변경 영향 | 매핑/검증 로직 영향 | 구조 변경 시 기준 | 구조 변경 시 파이프라인/웹 영향 |

### 6.5 데이터 구조 불일치/주의점
- config.yaml의 매핑 규칙이 변경될 경우, 기존 budget_db.json_backup과 구조가 달라질 수 있음
- budget_db.json_backup 내 일부 필드(예: `overview`, `effectiveness`, `kpi` 등)는 config.yaml에 명시적 매핑이 없을 수 있음
- 분석 결과(`analysis`) 구조는 template.json, config.yaml에 정의되어 있지 않으므로, 파이프라인 코드와 동기화 필요
- template.json의 구조가 최신이 아닐 경우, 실제 데이터와 불일치 가능

## 9. 구조화(structured) → 파싱(parsed) 과정 데이터 손실 원인 분석

### 현상 요약
- `과기부_1권_structured.json`에는 원본 PDF에서 추출된 상세 텍스트, 표, 페이지별 정보가 풍부하게 포함되어 있음
- `과기부_1권_parsed.json`(및 budget_db.json_backup)에는 많은 필드가 비어 있거나(null/빈값), 텍스트/표 데이터가 누락됨
- 실제로는 사업별 목적, 내용, 예산, 표 등 주요 정보가 대부분 사라짐

### 원인 분석
1. **구조화 단계(`json_structurer.py`)**
  - PDF → Raw JSON → 구조화 시, front/projects/tail로 논리적 구역만 나누고, 실제 텍스트/표는 각 구역별로만 저장
  - 사업별로 pages, text, tables가 projects[사업코드]에 잘 들어있음
2. **파싱 단계(`budget_parser.py`)**
  - 구조화된 JSON을 받아서, 각 사업별로 텍스트/표를 꺼내 header/summary/budget 등 주요 필드를 추출
  - 이때, 텍스트 블록에서 키워드(사업목적, 내용, 근거 등)로만 일부 내용을 추출하고, 나머지 텍스트/표는 버림
  - 표 추출 로직이 엄격(헤더 매칭, 키워드 기반)하여, 실제 표 구조가 다르거나 누락되면 값이 None/빈값으로 남음
  - 예산 등 숫자 추출도 헤더/키워드가 없으면 None 처리됨
  - 결과적으로, 구조화 단계의 rich data가 파싱 단계에서 대부분 소실됨

### 코드상 주요 문제점
- `budget_parser.py`의 `_extract_text_block`, `_extract_top_summary_tables`, `_extract_budget` 함수가 키워드/패턴 기반으로만 일부 정보만 추출
- 사업별 전체 텍스트/표를 보존하지 않고, 일부 필드만 남김(나머지는 버림)
- 구조화된 데이터의 text/tables 전체를 별도 필드로라도 보존하지 않음

### 개선 방향
1. **사업별 원본 텍스트/표 보존**
  - 파싱 결과에 `raw_text`, `raw_tables` 등으로 구조화 단계의 전체 텍스트/표를 함께 저장
2. **키워드 기반 추출 실패 시 fallback**
  - 사업 목적/내용/예산 등 주요 필드가 비어 있으면, 전체 텍스트 일부라도 fallback으로 채우기
3. **표 추출 로직 완화/다양화**
  - 표 헤더/구조가 다를 때도 최대한 값 추출, 실패 시 원본 표라도 보존
4. **파싱/구조화 단계 통합 검증**
  - 구조화→파싱 전후 데이터 비교 자동화, 누락/손실 경고 출력

### 결론
- 현재 파이프라인은 "구조화→파싱" 단계에서 정보가 대폭 소실되는 구조
- 파싱 결과에 원본 텍스트/표를 함께 보존하고, 추출 실패 시 fallback/경고를 추가해야 함
- 구조화 데이터의 rich data를 최대한 활용하는 방향으로 파싱 로직 개선 필요

## 7. path_config.py 적용 가이드 및 실제 적용 파일

### 7.1 적용 대상 및 수정 파일
- **적용 대상:**
  - database, config, output 등 주요 경로를 직접 지정하거나, os.path/Path로 직접 경로를 조립하는 모든 파이프라인/전처리 스크립트
  - 예시: `scripts/preProc/main_cli.py` 등
- **수정 필요 파일:**
  - 기존에 `self.project_root = ...`, `self.data_dir = ...` 등으로 경로를 직접 조립하는 부분이 있는 파일
  - `os.path`, `Path`로 직접 경로를 만드는 부분을 path_config의 상수로 대체

### 7.2 적용 방법
1. **import**
   ```python
   from scripts.utils.path_config import DATA_DIR, RAW_DIR, STRUCT_DIR, PARSE_RESULT_DIR, OUTPUT_DIR, CONFIG_DIR, DEFAULT_CONFIG_PATH, TEMPLATE_PATH
   ```
2. **기존 경로 조립 코드 대체**
   - 예시 (main_cli.py):
     ```python
     # 기존
     self.data_dir = self.project_root / "database"
     self.raw_dir = self.data_dir / "raw"
     ...
     # 변경 후
     from scripts.utils.path_config import RAW_DIR, STRUCT_DIR, ...
     self.raw_dir = RAW_DIR
     self.struct_dir = STRUCT_DIR
     ...
     ```
3. **폴더 생성 보장**
   - 파이프라인 실행 전 `ensure_pipeline_dirs()` 호출로 폴더 자동 생성
   - 예시:
     ```python
     from scripts.utils.path_config import ensure_pipeline_dirs
     ensure_pipeline_dirs()
     ```

### 7.3 적용 시 유의사항

- 신규 경로 추가 시 path_config.py에만 추가하면 전체 코드에서 일관성 유지 가능
